# Repository Guidelines

只保留那些不容易从代码或 README 直接看出来、但又很容易做错的约定。

- 不要用“防御性 fallback”或临时兜底去掩盖问题。先找根因；出错时通常应优先保留可见性，而不是把失败静默吞掉。
- 新功能默认按跨平台处理，至少同时考虑 macOS 和 Windows，不要顺手做成只适配当前机器。
- 同一组 provider、模型选项或展示名如果会在多个页面/弹窗复用，必须抽成共享定义；不要产生重复代码。
- 需求还在讨论或尚未验证完成时，不要把方案写成“已经实现”。必须明确区分：`已完成`、`已实现但未验证`、`建议方案`、`待确认`。
- 涉及外部 ASR 服务不稳定的修复时，除了错误提示本身，还必须同时检查三件事：HUD 可见性、录音是否停止/释放设备、失败后是否还能继续切换 provider 或重试。
- 提交代码时不要把本地工作笔记或调试目录混进去，例如 `.local-notes/`、`.claude/`、以及其他仅本机使用的未跟踪文件。
- `dist/` 是生成产物，不要手改。
- `docs/` 和 `swift_ui_references/` 只放参考资料，不要往里面塞生成文件。
- 当前没有成型的自动化测试基线。提交前至少跑一次 `pnpm build`；如果改到 `src-tauri/`，还要至少补一轮相关的 `cargo check`，最好再跑和改动直接相关的 Rust 测试。前端测试贴近源码放 `*.spec.ts`，Rust 测试放在 `src-tauri/src` 内的 `#[cfg(test)]`。
- 提交信息沿用 conventional commits，如 `feat:`、`fix:`、`chore:`。
